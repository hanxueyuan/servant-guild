# Architecture Decision Records (ADR)

本文档记录了 ServantGuild 系统的关键架构决策。

## ADR-001: 采用使魔团 (ServantGuild) 架构
- **状态**: 已接受
- **背景**: 单体架构难以扩展，无法实现多角色协作与自我进化。
- **决策**: 将系统拆分为多个自治智能体（使魔），通过协作完成任务。
- **后果**: 增加了系统复杂度，但实现了自治与隔离。

## ADR-002: 使用 Rust 作为主要开发语言
- **状态**: 已接受
- **背景**: 系统对性能、内存安全和并发处理有极高要求。
- **决策**: 全栈使用 Rust (1.87+)。
- **后果**: 开发门槛较高，但保证了系统稳定性和效率。

## ADR-003: 采用 Wasmtime 作为隔离运行时
- **状态**: 已接受
- **背景**: 需要轻量级、安全的沙盒环境来运行不可信的智能体代码，且支持热更新。
- **决策**: 使用 Wasmtime 运行 Wasm 模块，而非 Docker 容器。
- **后果**: 启动速度 < 5ms，资源开销极低，但需处理 Wasm 绑定复杂性。

## ADR-004: 引入共识引擎 (Consensus Engine)
- **状态**: 已接受
- **背景**: 自治系统需要防止单点故障或错误决策。
- **决策**: 引入议长使魔，对重大变更实行投票机制。
- **后果**: 决策延迟增加，但系统安全性大幅提升。

## ADR-005: 使用 PostgreSQL 作为主数据存储
- **状态**: 已接受
- **背景**: 需要处理结构化数据（用户、任务、日志）和复杂的查询关系。
- **决策**: 使用 PostgreSQL 存储核心业务数据，Redis 作为缓存。
- **后果**: 保证了数据的一致性与完整性。

## ADR-006: 实施审慎代理 (Prudent Agency) 策略
- **状态**: 已接受
- **背景**: 智能体拥有高权限，误操作后果严重。
- **决策**: 强制执行“审计优先、快照常备、回滚就绪”策略。
- **后果**: 所有 OS 变更操作变慢，但具备了极强的可恢复性。

## ADR-007: 以 GitHub 为基因库 (Gene Pool)
- **状态**: 已接受
- **背景**: 智能体需要存储、版本化和分发自己的代码。
- **决策**: 使用 GitHub 仓库作为代码和 Release 的存储后端。
- **后果**: 依赖外部服务，需处理 API Rate Limit，但复用了成熟的 DevOps 生态。

## ADR-008: 基于 Tokio 的异步编程模型
- **状态**: 已接受
- **背景**: 系统是 I/O 密集型（网络请求、文件操作）。
- **决策**: 全面采用 Tokio 异步运行时。
- **后果**: 提高了并发吞吐量。

## ADR-009: 使用 OpenAPI 定义接口契约
- **状态**: 已接受
- **背景**: 系统涉及多语言交互（Rust, Wasm, Python 工具）和前后端分离。
- **决策**: 使用 OpenAPI 3.1 规范定义 RESTful 接口。
- **后果**: 规范了交互协议，便于生成客户端代码。

## ADR-010: 采用主干开发 (Trunk-Based Development)
- **状态**: 已接受
- **背景**: 需要快速迭代和持续集成，避免长分支地狱。
- **决策**: 采用主干开发模式，小步提交，快速合并。
- **后果**: 依赖强大的自动化测试体系（CI）来保证主干稳定性。
