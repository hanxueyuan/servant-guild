# 架构技术方案复核与完善建议书

**评审对象**: `docs/design/ai_model_spec_zh.md` (v1.0.1)
**评审日期**: 2026-02-26
**评审结论**: **需要补充完善 (Needs Refinement)**

---

## 1. 技术可行性分析

### 1.1 "审慎代理"机制 (Prudent Agency)
*   **可行性**: 高。利用 Rust 的 `std::process::Command` 包装层或 FUSE 文件系统挂载点可以实现拦截。
*   **风险**: **Windows 系统还原点 API (VSS)** 调用复杂，且创建速度慢（秒级甚至分钟级），直接影响 `SNAPSHOTTING` 阶段的延迟。
*   **建议**:
    *   **分级快照策略**: 对文件修改使用轻量级文件备份（File Copy）；仅对驱动安装/注册表修改等系统级变更调用 VSS。
    *   **Windows VSS 集成**: 需明确依赖 `winapi` 或 `vswriter` crate。

### 1.2 数据库选型与加密
*   **可行性**: 高。`sqlx` + `aes-gcm` 是 Rust 生态成熟组合。
*   **缺失**: 密钥管理 (KMS) 方案。目前的文档未说明 AES 密钥存储在哪里（环境变量？配置文件？）。
*   **建议**: 增加 **密钥轮换 (Key Rotation)** 和 **信封加密 (Envelope Encryption)** 机制描述。

### 1.3 任务调度延迟
*   **目标**: < 50ms。
*   **挑战**: 若每次任务分发都需查询 PG + 解密，高并发下可能超标。
*   **建议**: 明确 Redis 缓存层不仅缓存 Agent 配置，还需缓存 **解密后的热数据**（需评估内存安全风险）或采用 **Read-Through** 策略。

## 2. 文档完善点

### 2.1 补充“密钥管理”章节
在“安全”部分增加密钥生命周期管理：
*   **主密钥 (Master Key)**: 由运维人员通过环境变量注入，不落盘。
*   **数据密钥 (Data Key)**: 每个 Agent 生成独立密钥，被主密钥加密后存入 DB。

### 2.2 细化“快照策略”
在“安全恢复模块”中区分快照级别：
*   **Level 1 (File)**: 单个文件备份（< 10ms）。适用于代码编辑、配置修改。
*   **Level 2 (System)**: OS 还原点（> 5s）。适用于软件安装、环境配置。

### 2.3 增加“可观测性”设计
目前仅提到“审计日志”。建议增加：
*   **Metrics**: Prometheus 指标（快照耗时、回滚成功率、任务吞吐量）。
*   **Tracing**: OpenTelemetry 链路追踪，关联 TaskID 与 AuditID。

## 3. 结论
文档整体架构清晰，但对 Windows 特有的系统还原机制及密钥管理的落地细节描述不足。建议针对上述 3 点进行增补。
